## Known Fragility

- **Type addition cascade**: Adding a field to a wide type (e.g., `DetailRecord`) requires updating every file that constructs objects of that type. Missing one construction site causes a build failure. The code-inspector agent exists specifically for this.
- **Export column drift**: `ExportMenu` and `MetricDrillDownModal` have explicit column mappings. New fields added to types are silently excluded from exports unless these components are updated.
- **BigQuery field casing**: A typo in a BigQuery field name (e.g., `advisor_join_Date__c` vs `Advisor_Join_Date__c`) causes silent data loss — the query succeeds but the field returns NULL.
- **Prisma lazy init** (`src/lib/prisma.ts`): Uses Proxy for lazy initialization to avoid build-time Prisma instantiation. Replacing with direct `new PrismaClient()` breaks Vercel builds.
- **SGA/SGM auto-scoping**: API routes automatically filter data for SGA/SGM roles using `sgaFilter`/`sgmFilter`. Forgetting this in a new route exposes other users' data.

## BigQuery Query Rules

- **Parameterization**: All queries must use `@paramName` syntax — never string interpolation or template literals in SQL strings
- **Field naming**: Case-sensitive, sourced from Salesforce field API names. Always verify against schema before use.
- **NULL handling**: COALESCE with sensible defaults — keep records, never filter NULLs to silently drop data
- **Deduplication flags**: `is_sqo_unique`, `is_joined_unique`, `is_primary_opp_record` exist in `vw_funnel_master`. Use the appropriate flag for volume counts to avoid double-counting.
- **No writes**: The app never writes to BigQuery — all data flows Salesforce → BigQuery via Data Transfer Service (6 jobs, every 6 hours)
- **View constants**: Use `FULL_TABLE`, `FORECAST_TABLE`, `MAPPING_TABLE`, `DAILY_FORECAST_VIEW` from `src/config/constants.ts` — do not hardcode view paths

## Postgres/Prisma Rules

- **ORM only**: All Postgres access goes through Prisma client (`src/lib/prisma.ts`) — no raw SQL, no `pg` driver, no `@vercel/postgres` direct queries (the package is installed but unused by the main app)
- **Migration discipline**: Schema changes require `prisma migrate`. Prisma generate runs as part of `npm run build` and `postinstall`.
- **Sessions**: NextAuth uses JWT strategy (24h maxAge) — there are no session tables in Postgres. `@auth/prisma-adapter` is installed but NOT imported. Do not add database session management.
- **User provisioning**: Users must be pre-created by an admin. Google OAuth checks for an existing `User` record and rejects unknown emails. Do not add auto-provisioning.
- **Connection handling**: `src/lib/prisma.ts` uses Proxy-based lazy init with Neon-specific SSL, pooler detection, and timeout configuration. Do not replace with `new PrismaClient()`.
- **Retry logic**: `src/lib/users.ts` wraps DB ops in `retryDatabaseOperation()` for Neon connection timeouts (3 retries, exponential backoff, excludes EPERM errors).
